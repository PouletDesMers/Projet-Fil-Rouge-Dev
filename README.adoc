== Introduction

CYNA est une plateforme SaaS de cybersécurité complète, composée de quatre composants :

* **API Go** — Backend REST (port 8080)
* **Node.js** — Serveur web + proxy + interface admin (port 3000)
* **PostgreSQL** — Base de données (port 5432)
* **Application mobile** — React Native / Expo (Android & iOS)

Ce guide vous accompagne pas à pas dans l'installation et la configuration complète de la plateforme, y compris l'application mobile.

=== Prérequis matériels

[cols="1,1", options="header"]
|===
| Composant | Minimum Recommandé
| **RAM** | 4 Go (8 Go recommandé)
| **CPU** | 2 cœurs (4 cœurs recommandé)
| **Espace disque** | 10 Go (20 Go avec backups)
| **OS** | Linux (Ubuntu 22.04+, Debian 12+, ou WSL2 sur Windows)
|===

== Étape 4 : Sauvegardes

=== Principe

Les sauvegardes utilisent `pg_dump` pour créer des fichiers `.sql` dans le volume Docker `backups_data`. Aucune configuration supplémentaire n'est nécessaire.

=== Types de sauvegarde

[cols="1,2,2", options="header"]
|===
| Type | Contenu | Usage
| `full` | Toutes les tables | Sauvegarde complète
| `data` | Toutes les tables sauf `api_logs` | Données métier
| `logs` | Uniquement `api_logs` | Audit et conformité
|===

=== Sauvegarde manuelle

. Ouvrir le panel admin : http://localhost:3000/backend/
. Aller dans **Sauvegardes**
. Cliquer sur **"Lancer la sauvegarde"**
. Le fichier `.sql` apparaît dans la liste

=== Sauvegarde automatique (planification)

==== Via l'interface

. Dans la section **Planification automatique**
. Choisir le type (Complète / Données / Logs)
. Définir l'intervalle (ex: `60` + `min` pour toutes les heures)
. Cliquer **"Ajouter"**

==== Via variable d'environnement

[source,env]
----
# Dans le fichier .env
BACKUP_INTERVAL_MINUTES=1440  # Toutes les 24h
----

Puis redémarrer :

[source,bash]
----
docker compose down && docker compose up -d
----

=== Restauration

. Dans la liste des sauvegardes, cliquer sur l'icône **restaurer** ↻
. Confirmer la restauration
. Le fichier SQL est injecté dans PostgreSQL via `psql`

[TIP]
====
La restauration écrase les données existantes. Faites une sauvegarde avant de restaurer.
====

=== Commandes manuelles (avancé)

[source,bash]
----
# Lister les fichiers de sauvegarde
docker compose exec api ls -lh /backups/pgdump/

# Restaurer manuellement
docker compose exec -T db psql -U postgres -d mydb < backup.sql

# Créer une sauvegarde manuelle via pg_dump
docker compose exec db pg_dump -U postgres -d mydb \
  --data-only --inserts --no-owner --no-acl \
  -f /tmp/backup_manuelle.sql

# Télécharger une sauvegarde
docker compose cp api:/backups/pgdump/full_20260427_123456.sql .
----

== Étape 5 : Logs

=== Triple stockage

Les logs API sont écrits simultanément dans 3 supports :

[cols="1,2,2", options="header"]
|===
| Stockage | Durée | Commande d'accès
| **RAM** (500 entrées) | Perdu au redémarrage | Interface admin
| **PostgreSQL** | 90 jours (purge auto) | Interface admin ou SQL
| **Fichier** (JSONL) | Illimitée | `docker compose exec api tail -f /var/log/api/api.log`
|==|

=== Consultation des logs fichier

[source,bash]
----
# En temps réel
docker compose logs -f api

# Dernières 50 lignes du fichier
docker compose exec api tail -50 /var/log/api/api.log

# Filtrer les erreurs
docker compose exec api grep ERROR /var/log/api/api.log

# Compter le nombre de logs
docker compose exec api wc -l /var/log/api/api.log
----

=== Purge des logs

[source,bash]
----
# Depuis l'interface : Sauvegardes → Logs API → "Vider"
# Ou en SQL directement :
docker compose exec db psql -U postgres -d mydb \
  -c "DELETE FROM api_logs;"
----

=== Configuration

==== Variables d'environnement mobile

Le fichier `.env` à la racine contient les variables de l'application mobile. Celles préfixées `EXPO_PUBLIC_` sont accessibles côté client.

[source,env]
----
# ── URL de l'API Go ──
# Émulateur Android       → http://10.0.2.2:8080/api
# Simulateur iOS          → http://localhost:8080/api
# Téléphone réel (Wi-Fi)  → http://<IP_LAN_DE_VOTRE_PC>:8080/api
EXPO_PUBLIC_API_URL=http://192.168.1.42:8080/api

# ── Stripe — clé publique uniquement ──
# Ne jamais mettre STRIPE_SECRET_KEY dans ce fichier
EXPO_PUBLIC_STRIPE_PUBLISHABLE_KEY=pk_test_votre_cle_publique
----

WARNING: N'utilisez jamais `STRIPE_SECRET_KEY` côté mobile. La clé secrète ne doit exister que dans le `.env` côté serveur.

==== Trouver l'adresse IP de votre machine

[source,bash]
----
# Linux / macOS
ip a | grep "inet " | grep -v 127.0.0.1

# Windows (PowerShell)
ipconfig | findstr "IPv4"

# Exemple de résultat : 192.168.1.42
# → définir EXPO_PUBLIC_API_URL=http://192.168.1.42:8080/api
----

=== Problème : les conteneurs ne démarrent pas

=== Problème : "port already allocated"

[source,bash]
----
# Vérifier quel processus utilise le port
sudo lsof -i :3000
sudo lsof -i :8080
sudo lsof -i :5432

# Modifier les ports dans docker-compose.yml si nécessaire
# Remplacer "3000:3000" par "3001:3000"
----

=== Problème : erreur 429 (Too Many Requests)

Le rate limiting est atteint. Attendre 60 secondes que le quota se réinitialise.

[source,bash]
----
# Réinitialiser le rate limiting en redémarrant l'API
docker compose restart api
----

=== Problème : base de données corrompue

[source,bash]
----
# Réinitialiser complètement (⚠️ perd toutes les données)
docker compose down -v
docker compose up -d
----

=== Problème : "Network request failed" sur l'application mobile

L'application ne peut pas contacter l'API.

[source,bash]
----
# 1. Vérifier que l'API tourne
curl http://localhost:8080/health

# 2. Vérifier EXPO_PUBLIC_API_URL dans .env
#    Téléphone réel    → IP LAN du PC (ex: 192.168.1.42)
#    Émulateur Android → 10.0.2.2
#    Simulateur iOS    → localhost

# 3. Vérifier que téléphone et PC sont sur le même Wi-Fi

# 4. Autoriser le port 8080 dans le pare-feu
sudo ufw allow 8080      # Linux
# Windows : Pare-feu Windows → Règles entrantes → Autoriser 8080
----

=== Annexe E : Variables d'environnement — Application Mobile

Ces variables sont définies dans le fichier `.env` à la racine du projet. Le préfixe `EXPO_PUBLIC_` est obligatoire pour qu'elles soient accessibles depuis le code React Native.

[cols="1,2,2", options="header"]
|===
| Variable | Description | Exemple
| `EXPO_PUBLIC_API_URL` | URL complète de l'API Go, avec `/api` | `http://192.168.1.42:8080/api`
| `EXPO_PUBLIC_STRIPE_PUBLISHABLE_KEY` | Clé publique Stripe (pk_test_...) | `pk_test_abc123`
|===

[NOTE]
====
Les valeurs de `EXPO_PUBLIC_API_URL` varient selon l'environnement :

* **Téléphone réel** → `http://<IP_LAN_DU_PC>:8080/api`
* **Émulateur Android** → `http://10.0.2.2:8080/api`
* **Simulateur iOS** → `http://localhost:8080/api`

Après toute modification du `.env`, relancez avec `npx expo start --clear`.
====

WARNING: Ne jamais ajouter `STRIPE_SECRET_KEY` dans le `.env` mobile. Cette clé ne doit exister que côté serveur.
